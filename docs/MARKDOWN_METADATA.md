# Markdown Metadata Format

This document defines the planned Markdown metadata format for `storemeta`.

The goal is simple: users should be able to manage App Store Connect and Google Play listing text with readable `.md` files instead of YAML.

## Status

This is the implementation contract for upcoming Markdown metadata support.

Current released versions may still use YAML as the implemented default. Do not document Markdown as fully available in release notes until the implementation and tests listed in [`TODO.md`](TODO.md) are complete.

## Position

Markdown should become the default metadata authoring format.

YAML remains supported as an alternative for users who prefer structured files or automation-generated metadata.

No mixed mode:

- `metadata.format: markdown` reads and writes `.md` files.
- `metadata.format: yaml` reads and writes `.yml` or `.yaml` files.
- A single configured app should not mix Markdown and YAML metadata files.

The default generated by `storemeta init` should be Markdown after this feature ships.

## Config

Default Markdown metadata config:

```yaml
metadata:
  baseDir: metadata
  format: markdown
```

Alternative YAML metadata config:

```yaml
metadata:
  baseDir: metadata
  format: yaml
```

Config validation rules:

- `metadata.format` accepts only `markdown` or `yaml`.
- New projects generated by `storemeta init` use `markdown`.
- Existing YAML projects continue to work when `metadata.format: yaml` is set.
- If `metadata.format` is omitted in a future config version, it should resolve to `markdown`.
- In config version `1`, omitting `metadata.format` should either keep the existing validation behavior or be migrated deliberately in a separate config-version decision.

## Directory Layout

Markdown keeps the same platform and locale layout as YAML.

```text
metadata/
  apple/
    en-US.md
    tr.md
  google/
    en-US.md
    tr.md
```

The platform is resolved from the parent directory:

- `metadata/apple/*.md` maps to App Store Connect metadata.
- `metadata/google/*.md` maps to Google Play metadata.

The locale is read from frontmatter. The filename must match the normalized `locale` value plus `.md`.

Valid:

```text
metadata/apple/en-US.md
metadata/google/tr.md
```

Invalid:

```text
metadata/apple/english.md
metadata/google/tr-TR.md
```

The second example is invalid when frontmatter says `locale: tr`. If frontmatter also says `locale: tr-TR`, validation may normalize it according to the existing locale rules before comparing the filename.

## File Discovery

When `metadata.format: markdown` is configured:

- Only files ending in `.md` are considered metadata files.
- `.yml` and `.yaml` files inside metadata platform directories are format mismatches.
- Format mismatches should fail `storemeta validate`.
- Push commands should fail before remote writes if format mismatches are present.
- Hidden files such as `.DS_Store` are ignored.
- Subdirectories under `metadata/apple` or `metadata/google` are ignored unless a future layout explicitly supports them.

When `metadata.format: yaml` is configured:

- Only `.yml` and `.yaml` files are considered metadata files.
- `.md` files inside metadata platform directories are format mismatches.

## File Structure

Each Markdown metadata file has two parts:

1. YAML frontmatter for routing and metadata.
2. Markdown sections for store listing fields.

```md
---
locale: en-US
store_locale: en-US
source: local
---

# App Store Listing

## App Name

Example App

## Description

Long store description.
```

Required frontmatter:

- `locale`

Optional frontmatter:

- `store_locale`
- `source`
- `fetched_at`
- `platform`

`platform` is optional because the platform directory is authoritative. If present, it must match the parent directory.

Frontmatter rules:

- Frontmatter must be the first content in the file.
- Frontmatter must start with `---` on the first line and end with `---` on its own line.
- Frontmatter is parsed as YAML.
- Unknown frontmatter keys are allowed only if they are namespaced with `x_`.
- `locale`, `store_locale`, `source`, `fetched_at`, and `platform` must be strings when present.
- Empty frontmatter values fail validation.
- `platform: app_store` is not accepted in the new format; use `platform: apple`.
- `platform: google_play` is not accepted in the new format; use `platform: google`.

Reserved frontmatter keys:

| Key | Required | Meaning |
| --- | --- | --- |
| `locale` | yes | Local metadata locale used by storemeta. |
| `store_locale` | no | Remote store locale if different from local locale. |
| `platform` | no | Optional duplicate of the platform directory, either `apple` or `google`. |
| `source` | no | Provenance such as `local`, `pulled`, or `localized_from_en-US`. |
| `fetched_at` | no | ISO timestamp written by pull commands. |

Implementation metadata that should not be uploaded, such as remote IDs or version state, should live in frontmatter under `x_` keys if needed.

## Apple Fields

Apple Markdown files use these headings:

```md
---
locale: en-US
store_locale: en-US
source: local
---

# App Store Listing

## App Name

Example App

## Subtitle

Short subtitle

## Promotional Text

Optional promotional text.

## Description

Long App Store description.

## Keywords

keyword one,keyword two,keyword three

## What's New

Release notes for this version.

## Support URL

https://example.com/support

## Marketing URL

https://example.com

## Privacy Policy URL

https://example.com/privacy
```

Heading to internal field mapping:

| Markdown heading | Internal field |
| --- | --- |
| `App Name` | `app_name` |
| `Subtitle` | `subtitle` |
| `Promotional Text` | `promotional_text` |
| `Description` | `description` |
| `Keywords` | `keywords` |
| `What's New` | `whats_new` |
| `Support URL` | `support_url` |
| `Marketing URL` | `marketing_url` |
| `Privacy Policy URL` | `privacy_policy_url` |

Apple heading aliases:

| Alias | Canonical heading |
| --- | --- |
| `Name` | `App Name` |
| `Promotional text` | `Promotional Text` |
| `Release Notes` | `What's New` |
| `Whats New` | `What's New` |
| `What is New` | `What's New` |
| `Privacy URL` | `Privacy Policy URL` |

Apple validation should reuse the existing Apple metadata schema after parsing. Store-specific length limits should be added when implemented locally.

## Google Fields

Google Markdown files use these headings:

```md
---
locale: en-US
store_locale: en-US
source: local
---

# Google Play Listing

## Title

Example App

## Short Description

Short Google Play description.

## Full Description

Long Google Play description.

## Video

https://example.com/video
```

Heading to internal field mapping:

| Markdown heading | Internal field |
| --- | --- |
| `Title` | `title` |
| `Short Description` | `short_description` |
| `Full Description` | `full_description` |
| `Video` | `video` |

Google heading aliases:

| Alias | Canonical heading |
| --- | --- |
| `Name` | `Title` |
| `App Name` | `Title` |
| `Short description` | `Short Description` |
| `Description` | `Full Description` |
| `Full description` | `Full Description` |
| `Promo Video` | `Video` |

Google validation should reuse the existing Google metadata schema after parsing, including length checks:

- `title`: 30 characters or fewer
- `short_description`: 80 characters or fewer
- `full_description`: 4000 characters or fewer

## Parsing Rules

- Frontmatter is required.
- `locale` is required.
- Heading names are case-insensitive after trimming whitespace.
- Unknown field headings should fail validation.
- Duplicate field headings should fail validation.
- Empty sections are treated as missing fields.
- The top-level `# App Store Listing` or `# Google Play Listing` heading is ignored by the parser.
- Store text is uploaded as plain text. Markdown bullets and line breaks are preserved as characters; Markdown is not rendered to HTML.
- Code fences should not be required around section content.
- Field sections must use level-two headings: `## Field Name`.
- Level-three and deeper headings inside field content are preserved as content.
- Content before the first recognized level-two field heading is ignored only if it is the top-level title area. Any other freeform content should fail validation.
- Leading and trailing blank lines inside section bodies are trimmed.
- Internal blank lines are preserved.
- Line endings are normalized to `\n`.
- HTML comments are preserved as content unless a later implementation explicitly strips them.
- Markdown links are preserved as literal Markdown text.
- YAML frontmatter comments are not preserved when files are rewritten by pull.

Recommended parser algorithm:

1. Read the file as UTF-8.
2. Split and parse frontmatter.
3. Scan Markdown line by line.
4. Ignore one top-level `# ...` title before the first field.
5. Treat each `## ...` heading as a field boundary.
6. Canonicalize the heading using the platform heading map and aliases.
7. Preserve every non-boundary line as plain text for the current field.
8. Trim section body edges and drop empty fields.
9. Build the existing internal Apple or Google metadata document.
10. Run existing platform validation.

## Serialization Rules

Generated Markdown files should be stable so diffs stay readable.

- Use UTF-8.
- Use `\n` line endings.
- End each file with one trailing newline.
- Write frontmatter first.
- Write headings in the documented canonical order.
- Do not wrap long lines.
- Do not use code fences for field values.
- Do not include empty optional fields when writing pulled metadata, unless generated by `scaffold`.
- `scaffold` may include empty headings to show the expected shape.
- Escape neither URLs nor commas; store text should remain directly editable.

## Pull Behavior

When `metadata.format: markdown` is configured:

- `storemeta metadata pull` writes `.md` files.
- Existing `.md` files may be overwritten by pull, matching current metadata pull behavior.
- Generated files should use stable heading order.
- Long text fields should be written as plain Markdown section bodies.
- Pull should include `fetched_at` in frontmatter.
- Pull may include `store_locale` when the remote locale differs from normalized `locale`.
- Pull should not include secrets or credential-derived values.

Apple pull target:

```text
metadata/apple/en-US.md
```

Google pull target:

```text
metadata/google/en-US.md
```

## Push Behavior

When `metadata.format: markdown` is configured:

- `storemeta metadata push` reads only `.md` metadata files.
- YAML metadata files in the same app metadata directory are reported as format mismatches.
- Parsed Markdown is mapped into the existing Apple and Google metadata document models before API upload.
- `--dry-run` should show the same per-locale behavior as YAML.
- Push must validate every selected Markdown metadata file before making any remote write.
- Push should ignore frontmatter provenance fields such as `source`, `fetched_at`, and `x_*`.

## Diff Behavior

When `metadata.format: markdown` is configured:

- `storemeta metadata diff` compares parsed Markdown documents, not raw Markdown formatting.
- Heading capitalization differences should not create diffs if they map to the same canonical field.
- Frontmatter provenance fields such as `source`, `fetched_at`, and `x_*` should not create store metadata diffs.
- Meaningful listing text changes should be reported by platform, locale, and field.
- Missing local files should be reported as missing locales.
- Extra local files should be reported as extra locales.
- Format mismatches should fail before comparison.

## Validate Behavior

`storemeta validate` should check Markdown metadata files for:

- valid frontmatter
- required locale
- filename and locale consistency
- allowed headings for the selected platform
- duplicate headings
- unknown frontmatter keys unless they use `x_`
- `platform` mismatch between frontmatter and directory
- platform-specific metadata schema
- Google Play length limits
- format mismatches when files from the other metadata format exist in the selected platform directory

## Error Messages

Errors should include enough context for users to fix the file without reading source code.

Good error targets:

```text
metadata.apple.en-US.frontmatter.locale
metadata.apple.en-US.heading.Description
metadata.google.tr.Short Description
```

Example messages:

```text
metadata.apple/en-US.md: missing required frontmatter field "locale"
metadata.apple/en-US.md: unknown heading "Tagline"; expected App Name, Subtitle, Promotional Text, Description, Keywords, What's New, Support URL, Marketing URL, or Privacy Policy URL
metadata.google/tr.md: duplicate heading "Full Description"
metadata.google/tr.md: short_description must be 80 characters or fewer
metadata.apple/en-US.md: platform frontmatter says "google" but file is under metadata/apple
metadata.google/tr.yml: YAML metadata file found while metadata.format is markdown
```

Remote writes must not start if any selected metadata file has a parse or validation error.

## Migration From YAML

Migration can be handled manually at first:

1. Set `metadata.format: markdown`.
2. Convert `metadata/apple/*.yml` to `metadata/apple/*.md`.
3. Convert `metadata/google/*.yml` to `metadata/google/*.md`.
4. Run `storemeta validate`.
5. Run `storemeta metadata push --dry-run` before any real push.

A future helper command may automate this, for example:

```bash
storemeta metadata convert --from yaml --to markdown
```

That command is not required for the first Markdown support implementation.

## Non-Goals

- No mixed Markdown/YAML metadata mode.
- No folder-per-locale Markdown layout in the first implementation.
- No Markdown-to-HTML rendering.
- No screenshots metadata inside Markdown listing files.
- No support for arbitrary localized sections outside the platform heading maps.
- No support for editing Apple review information or pricing in this format.
